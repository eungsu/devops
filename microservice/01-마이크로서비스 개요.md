# 마이크로서비스와 스프링부트

## 마이크로서비스
### 마이크로서비스란 무엇인가?
마이크로서비스는 큰 응용 프로그램을 작고 독립적인 서비스들로 분할하는 소프트웨어 개발 방식입니다. 이러한 각각의 서비스는 특정한 비즈니스 기능을 수행하며, 다른 서비스와 네트워크를 통해 통신할 수 있습니다. 

각 마이크로서비스는 독립적으로 배포하고 확장할 수 있으며, 별도의 개발 팀이 관리할 수 있습니다. 이러한 특성은 소프트웨어 개발과 배포의 복잡성을 줄이는 데 도움이 되며, 장애가 발생했을 때 해당 서비스만 고치면 되므로 전체 시스템의 안정성을 향상시키는 데 기여합니다.

마이크로서비스 아키텍처는 다음과 같은 주요 원칙에 기반합니다:

1. **분리 가능성** : 각 서비스는 독립적으로 배포되고 실행될 수 있어야 합니다.
2. **단일 책임 원칙** : 각 서비스는 하나의 비즈니스 기능만을 담당해야 합니다.
3. **분산 개발** : 서로 다른 개발 팀이 각각의 서비스를 독립적으로 개발하고 유지보수할 수 있어야 합니다.

마이크로서비스는 복잡한 애플리케이션을 빠르게 개발하고 변경하기 위한 강력한 도구로, 많은 기업들이 이 아키텍처를 채택하고 있습니다. 그러나 이 아키텍처는 도입과 운영에서의 복잡성, 데이터 관리, 네트워크 지연, 보안 등의 과제를 안고 있습니다. 따라서 해당 조직의 특성과 필요성에 따라 적절한 아키텍처를 선택하는 것이 중요합니다.

### 마이크로서비스의 특징
마이크로서비스 아키텍처는 다음과 같은 주요 특징을 가지고 있습니다:

1. **분산 개발**: 각각의 마이크로서비스는 독립적으로 개발되고 배포될 수 있습니다. 이는 서로 다른 개발 팀이 서로 다른 기술 스택을 사용할 수 있으며, 각 서비스의 업데이트 주기가 다를 수 있음을 의미합니다.

2. **높은 응집력과 낮은 결합도**: 마이크로서비스는 '단일 책임 원칙'을 따르는 것이 일반적입니다. 이는 각 서비스가 한 가지의 기능에 집중하고, 이 기능이 변경되어야 할 때 해당 서비스만 변경되면 된다는 의미입니다. 이로 인해 서비스 간의 결합도는 낮아집니다.

3. **독립적인 배포와 확장성**: 마이크로서비스는 각각 독립적으로 배포되고 확장될 수 있습니다. 이는 각 서비스가 독립적으로 운영되므로 특정 서비스에 문제가 생겨도 전체 시스템에 영향을 미치지 않습니다.

4. **데이터 격리**: 각 마이크로서비스는 자체 데이터베이스를 갖는 것이 일반적입니다. 이는 데이터가 서비스 간에 공유되지 않고 각 서비스가 자신의 데이터에 대해 완전한 제어권을 갖는다는 것을 의미합니다.

5. **통신**: 마이크로서비스는 네트워크를 통해 통신합니다. 일반적으로 이러한 통신은 HTTP/REST 또는 비동기 메시징을 사용하여 수행됩니다.

이러한 특징들은 마이크로서비스가 복잡한 시스템을 유연하게 다루고 빠르게 개선할 수 있도록 해주지만, 동시에 데이터 관리, 네트워크 지연, 분산 시스템의 복잡성 등 새로운 도전 과제를 제시합니다. 따라서 마이크로서비스 아키텍처는 적절한 계획과 준비가 필요합니다.

### 마이크로서비스 아키텍처의 구성요소
마이크로서비스 아키텍처에서 자주 사용되는 몇 가지 주요 구성요소와 패턴들은 다음과 같습니다:

1. **서비스**: 마이크로서비스 아키텍처의 핵심은 작은, 독립적으로 배포되는 서비스입니다. 각 서비스는 특정 비즈니스 기능을 수행하며, 각각이 자신의 코드베이스를 가지고 있습니다.

2. **API 게이트웨이**: 클라이언트와 서비스 간의 통신을 쉽게하기 위해, API 게이트웨이는 클라이언트의 요청을 적절한 서비스로 라우팅하는 역할을 합니다.

3. **서비스 디스커버리**: 이 구성 요소는 서비스들이 서로를 찾을 수 있도록 돕습니다. 각 서비스는 서비스 디스커버리에 자신의 위치를 등록하고, 다른 서비스가 필요할 때 서비스 디스커버리를 통해 그 위치를 찾습니다.

4. **로드 밸런서**: 로드 밸런서는 들어오는 요청을 서비스 인스턴스 간에 분산시키는 역할을 합니다. 이는 시스템의 확장성을 향상시키고 가용성을 높이는 데 도움이 됩니다.

5. **분산 트레이싱**: 마이크로서비스에서 요청은 여러 서비스를 거쳐 처리됩니다. 따라서 오류를 디버깅하거나 성능 문제를 파악하기 위해서는 분산 트레이싱이 필요합니다.

6. **서킷 브레이커**: 일부 서비스에 문제가 발생한 경우, 이 문제가 전체 시스템에 영향을 미치는 것을 막기 위해 서킷 브레이커 패턴이 사용됩니다. 서킷 브레이커는 서비스 간의 통신을 모니터링하고, 특정 서비스에서 오류가 반복적으로 발생할 경우 그 서비스로의 요청을 일시적으로 중단합니다.

7. **컨테이너화와 오케스트레이션**: 마이크로서비스는 종종 컨테이너 내에서 실행되며, 컨테이너 오케스트레이션 도구 (예: Kubernetes)는 이러한 컨테이너를 관리하고 확장하는 데 사용됩니다.

마이크로서비스 아키텍처는 이러한 구성요소와 패턴을 적절히 조합하여 복잡한 애플리케이션을 관리하고 확장하는 유연성을 제공합니다. 하지만 이러한 패턴들을 도입하고 관리하는 것은 복잡성을 증가시킬 수 있으므로, 적절한 계획과 실행이 필요합니다.

## 스프링부트와 스프링 클라우드
스프링 부트(Spring Boot)와 스프링 클라우드(Spring Cloud)는 자바(Java) 개발자들에게 강력한 프레임워크를 제공하며, 마이크로서비스 아키텍처를 구현하는 데 주로 사용됩니다.

**스프링 부트**는 자바 애플리케이션 개발을 간소화하는 프레임워크입니다. 스프링 부트는 단독 실행 가능한 애플리케이션을 생성하는 기능을 제공하며, '의존성 관리', '자동 구성' 등의 기능을 제공해 개발자가 빠르고 쉽게 애플리케이션을 구축하고 배포할 수 있도록 돕습니다.

스프링 부트의 주요 기능들:

- 자동 구성: 스프링 부트는 애플리케이션에 필요한 대부분의 기본 설정을 자동으로 처리해줍니다.
- 내장 서버: 애플리케이션을 별도의 WAR 파일로 패키징하고 서버에 배포할 필요 없이, JAR 파일 하나로 실행 가능한 애플리케이션을 만들 수 있습니다.
- 의존성 관리: 스프링 부트는 프로젝트에 필요한 의존성을 쉽게 관리할 수 있는 환경을 제공합니다.

**스프링 클라우드**는 마이크로서비스 아키텍처를 구현하는 데 필요한 통합 패턴을 제공합니다. 분산 시스템(마이크로서비스)의 공통 패턴을 간소화하고 표준화하는 도구로, 서비스 발견, 설정 관리, 회로 차단, 인텔리전트 라우팅, 마이크로 프록시 등과 같은 서비스를 제공합니다.

스프링 클라우드의 주요 기능들:

- 서비스 디스커버리: Eureka 서버와 같은 도구를 사용해 동적으로 서비스를 찾아내는 기능
- 클라우드 컨피그: 분산 시스템에서 외부 설정을 관리하는 기능
- 클라우드 버스: 분산 시스템에서의 이벤트와 상태 정보를 전파하는 기능
- 회로 차단: Hystrix와 같은 라이브러리를 통해 서비스 간의 통신에 대한 장애를 격리하는 기능

스프링 부트와 스프링 클라우드는 함께 사용될 때 강력한 마이크로서비스 구현 도구를 제공합니다. 
이 두 프레임워크를 사용하면 개발자는 마이크로서비스의 복잡성을 추상화하고 서비스 간의 통신, 서비스 디스커버리, 구성 관리 등을 효과적으로 처리할 수 있습니다.
